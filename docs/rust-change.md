# Rust engine change — Points-at-stake review queue (PRD 7a)

A new **read-only** backend method that returns the current due *review* queue
re-ordered by a **points-at-stake** score so the topics a student is weakest at
surface first:

```
points_at_stake = topic_weight x student_weakness
```

It changes **no** FSRS intervals, scheduling, or stored card data — it only
reads existing FSRS memory state and reorders ids.

## Why this must live in Rust (rslib)

- **The data and the maths already live here.** The due-review queue is built by
  the Rust queue builder (`rslib/src/scheduler/queue/`), and FSRS retrievability
  is computed by the bundled `fsrs` crate from per-card memory state managed in
  Rust. This change reuses the *exact* same call the browser/stats use
  (`FSRS::current_retrievability_seconds`), so the score can never drift from the
  engine's own notion of retrievability.
- **Write once, get every front-end for free.** Exposing it as a protobuf RPC on
  `SchedulerService` means desktop (Python), iOS (via the xcframework) and web
  all call the identical implementation. A Swift/JS reimplementation would
  duplicate FSRS math + queue/tag/bury logic and require re-validation per
  platform.
- **Avoids N+1 bridge traffic.** Scoring touches every due card, its note tags
  and its memory state. Doing that in Rust is one in-process pass; doing it in a
  front-end would mean marshalling every card + note across the IPC bridge.

## Design

New RPC on `SchedulerService`: **`GetPointsAtStakeQueue`** — service index
**13**, method index **39** (appended after `FuzzDelta`, so existing method
indices are unchanged).

Core entry point: `Collection::points_at_stake_queue(PointsAtStakeRequest)` in
`rslib/src/scheduler/points_at_stake.rs`.

1. **Gather the queue (read-only).** `Collection::due_review_card_ids()` builds
   the normal study queue and collects the `Review` main-queue entries in queue
   order. Building the queue does not write to the DB or create undo entries, so
   it respects active decks, daily review limits and sibling burying — i.e. the
   exact cards the user would review today.
2. **Per-card topic.** From the card's note tags, take the first tag under the
   configurable prefix (default `MCAT::`, ASCII case-insensitive) and use its
   **top component** (the level immediately under the prefix), e.g.
   `MCAT::Biochem::Enzymes` → `Biochem`. Notes with no matching tag bucket into
   `untagged`.
3. **Per-card retrievability.** `fsrs.current_retrievability_seconds(memory_state,
   seconds_since_last_review, decay)` — identical to `browser_table` / stats.
   Cards without FSRS memory state contribute `None`.
4. **Per-topic weakness.** `weakness = 1 - mean(retrievability)` over that
   topic's cards that have memory state (lower retrievability ⇒ weaker ⇒ higher
   weakness). A topic with no memory-state cards gets weakness `0`.
5. **Topic weight (simple + documented).** Default = `1.0` for every topic, so
   `points_at_stake == weakness` and the weakest topics sort first. With
   `weight_by_topic_size = true`, weight = the topic's share of the queue
   (`card_count / total_due`), favouring larger weak topics.
6. **Order.** Sort by descending `points_at_stake`, breaking ties by ascending
   card id for deterministic output. Returns the ordered cards (each with its
   topic, score and own retrievability) plus per-topic summaries for
   transparency/demo.

**Read-only / undo-safe guarantee.** The implementation only calls
`storage.get_card`, `storage.get_note`, `timing_today` and the FSRS
retrievability function. There is no `update_card`, no transaction and no undo
entry, so answer + undo and FSRS scheduling are provably unaffected (covered by
both Rust and Python tests).

## Merge-difficulty note

**Low — almost entirely additive.** `rslib` was unmodified upstream (commit
`b00308e5`), and this patch:

- Appends the proto rpc at the **end** of `SchedulerService` and the messages at
  the **end** of `scheduler.proto`, so every existing method index is preserved
  (the new method is 39; 0–38 are untouched). Wire indices regenerate
  consistently for both the Rust dispatch and the Python `_run_command`.
- Adds one new self-contained module file (no conflict surface) and only *adds*
  lines to existing files (one `mod` line, one queue helper method, one service
  method) — no existing logic is rewritten.

The only realistic upstream-rebase friction would be if upstream re-orders
`SchedulerService` methods, changes the queue builder's `MainQueueEntry` /
`get_queues` surface, or changes `fsrs::current_retrievability_seconds` — all
stable APIs today. iOS needs no Swift changes here; rebuilding the xcframework
from this branch will expose the new RPC automatically.
